Greentic · cloud-update-demo

Update a running environment from a server it doesn't trust

Start an environment on your own machine. Publish a new version to a plan server on the internet. The runtime discovers it, verifies the signature against a key it already holds, and moves traffic onto it — by itself, with no restart, and without ever trusting the server.

What you're about to run

Everything runs on your PC. The only thing in the cloud is a plan store: a dumb server that holds signed bytes and answers GETs. It has no signing key, so it cannot invent an update — it can only fail to hand one over, which your runtime survives. 

   your operator key                       (minted on YOUR machine by `op env init`,
        │  signs                           trusted by YOUR env, never uploaded)
        ▼
  ┌────────────────┐   sign + upload    ┌──────────────────────┐
  │  op updates    │ ─────────────────▶ │   plan server        │
  │  publish       │                    │   stores bytes.      │
  └────────────────┘                    │   holds no keys.     │
                                        └──────────────────────┘
                                                   ▲
                                    anonymous GET  │  (poll every 60s)
                                                   │
  ┌──────────────────────────────────────────────────────────┐
  │  greentic-start  ── verify signature vs trust root       │
  │                  ── snapshot → apply → roll back on fail │
  └──────────────────────────────────────────────────────────┘
             v1  (webchat)   ⇄   v2  (webchat + telegram)

You'll flip that environment between v1 and v2 as many times as you like, from the server, while it keeps serving.

Before you start

WhatWhy
greentic-deployer ≥ 1.1.10 The only thing you install. It signs and publishes plans.
bash, curl, tar, python3 Already on every Linux and macOS box.

Works on Linux and macOS (Intel and Apple Silicon) with the stock toolchain — no Homebrew, no coreutils. greentic-start is not a prerequisite: the demo downloads the released runtime it needs and checks it against the published .sha256 sidecar before running a byte of it.

install & clone 
$ cargo binstall greentic-deployer@1.1.10

$ git clone https://github.com/greenticai/greentic-cloud-update-demo
$ cd greentic-cloud-update-demo

Nothing here touches your real setup.

Every command runs against an isolated ./home inside the checkout, so your real ~/.greentic is never read or written. ./demo.sh clean removes all of it.

The short version

One command runs the entire story unattended — it applies v1, starts the runtime, publishes v2, waits for the runtime to converge, then publishes a binary update and watches the runtime replace its own executable. It takes about three minutes and asserts every claim it makes.

the whole thing 
$ ./demo.sh

If you'd rather drive it yourself — which is the point of the rest of this page — use the step commands below instead.

Driving it yourself

  1. Create the environment and apply v1

    This claims a private namespace on the plan server, mints your operator key, seeds its public half into the environment's trust root, and deploys v1 — and subscribes to the update channel, because the manifest declares one.

    terminal 1 
    $ ./demo.sh env

  ✓ namespace demo-c799f11df600c8d7 — expires in 24h
  ✓ manifests/v1.gen.json (carries the `updates` block) and v2.gen.json
▸ op env init — mints your operator key, seeds the env's trust root
    operator_key_id=13795242d8f8f13909d105efc2842cd7
▸ op env apply — deploys v1 AND subscribes to the channel
  ✓ v1 is live:
    sequence 1   sha256:6e6f24f5…   [app-webchat-bot]

    Remember that operator_key_id. Every plan you publish below is signed with it, and the runtime will refuse anything that isn't.

  2. Start the runtime

    It serves v1 and starts polling the plan server every 60 seconds. Leave this terminal open — its log is the demo.

    terminal 1 — keep it running 
    $ ./demo.sh serve

  3. Publish v2 to the server

    In a second terminal. This signs the v2 manifest in memory, reads the next sequence number off the server, and uploads the bytes. Your private key never leaves the machine.

    terminal 2 
    $ ./demo.sh switch v2

▸ op updates publish — next sequence from the server, sign in memory, upload
    sequence=1  key_id=13795242d8f8f13909d105efc2842cd7
  ✓ uploaded. Nothing was pushed at the runtime.

    Now do nothing. Nobody told the runtime anything.

  4. Watch it converge on its own

    Within one poll interval (≤60s) terminal 1 fetches the plan, verifies the signature against its own trust root, snapshots, applies, and moves traffic. No restart.

    terminal 1 — the runtime's own log 
    update-notify: staged update plan for env `local`
  deploy-bundle  updatedemo  update  digest sha256:6e6f24f5 → sha256:feb45faf
update-notify: applied update plan `01KXCW2Q…` to env `local`
update-poll: env `local` plan sequence 1 -> 200 (applied)
runtime-config reloaded: 1 → 1 deployment(s), 1 → 1 revision(s)

  5. Check what's live

    status shows what your runtime is actually serving next to what the server is holding. The pack list is the giveaway.

    terminal 2 
    $ ./demo.sh status

▸ what your runtime is actually serving
    sequence 2   sha256:feb45faf…   [app-webchat-bot, messaging-telegram]
  v1 = [app-webchat-bot]   v2 = [app-webchat-bot, messaging-telegram]
▸ what the plan server is holding
    sequence=1  plan_sha256=f69d93e2f4f165a6…
▸ runtime is up on :8080

  6. Switch back — and forth, as often as you like

    Publish v1 again and the environment returns to webchat-only, live, in under a minute.

    terminal 2 
    $ ./demo.sh switch v1
$ ./demo.sh status
    sequence 3   sha256:6e6f24f5…   [app-webchat-bot]

$ ./demo.sh switch v2
$ ./demo.sh status
    sequence 4   sha256:feb45faf…   [app-webchat-bot, messaging-telegram]

    Going back to v1 is not a downgrade.

    Anti-rollback protects the plan sequence, which only ever grows — 1, 2, 3, 4. The content a plan points at is free to move in either direction. That's why you can flip v1 ⇄ v2 forever while replay of an old signed plan stays impossible.

When you're done

cleanup 
# Ctrl-C the runtime in terminal 1, then:
$ ./demo.sh clean

What this actually proves

ClaimWhat enforces it
The server can't forge an update Plans are signed by your operator key. The server stores bytes and has no key. A plan it invented would fail verification against your trust root.
An attacker on the wire can't either Plan reads are anonymous and unauthenticated on purpose. The signature is the trust anchor — not TLS, not the origin.
You can't be replayed an old version Every plan carries a monotonic sequence, and the anti-rollback check reads it from the signed plan, not from anything the server says.
Two publishers can't fork the channel The store enforces strict monotonicity inside a Durable Object, so the read-modify-write is serialized. The loser of a race gets a 409, never a silent fork.
Nothing is pushed at your runtime It polls. Discovery is not authority: an environment that never declared an updates block fetches nothing, ever.
A bad update rolls back Apply runs under a pre-apply snapshot with automatic rollback and post-apply verification.

If something goes wrong

“port 8080 is already in use”

Something else holds the runtime's port. Free it, or pick another:

bash 
$ RUNTIME_PORT=8090 ./demo.sh serve

“a greentic-start is already running”

The demo refuses to start while another runtime is alive, and it means it. A stray runtime pointed at this checkout's HOME polls the same channel — and if a plan carries a binary, it will swap its own executable, which may be the one on your PATH. Stop it (kill <pid>) and re-run.

Your namespace expired

Namespaces on the shared server live for 24 hours. Just create the environment again — you'll get a fresh one:

bash 
$ ./demo.sh env

Why is the upload endpoint open to anyone?

Because it costs an attacker nothing worth having. Guess someone's namespace and the worst you can do is deny them an update: their runtime will reject any plan not signed by a key its trust root already carries. That is the whole point — the server is not a trusted party, and the demo is built so you can see that for yourself.

Run your own plan server

The server is a small Cloudflare Worker, and its source is in server/ in the same repo. Deploy your own and point the demo at it — nothing else about the run changes:

bash 
$ cd server && ./deploy.sh

$ SERVER=https://your-worker.workers.dev ./demo.sh env
