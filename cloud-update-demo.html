Greentic · cloud-update-demo

Update a running environment from a server it doesn't trust

Start an environment on your machine. Publish a new version to a plan server on the internet. The runtime finds it, verifies the signature against a key it already holds, and moves traffic onto it — by itself, with no restart, and without ever trusting the server.

Plain commands. No scripts to read, nothing hidden in a wrapper.

How it fits together

Everything runs on your machine. The only thing in the cloud is a plan store: a dumb server that holds signed bytes and answers GETs. It has no signing key, so it cannot invent an update — it can only fail to hand one over, which your runtime survives. 

   your operator key                       (minted on YOUR machine by `op env init`,
        │  signs                           trusted by YOUR env, never uploaded)
        ▼
  ┌────────────────┐   sign + upload    ┌──────────────────────┐
  │  op updates    │ ─────────────────▶ │   plan server        │
  │  publish       │                    │   stores bytes.      │
  └────────────────┘                    │   holds no keys.     │
                                        └──────────────────────┘
                                                   ▲
                                    anonymous GET  │  (poll every 60s)
                                                   │
  ┌──────────────────────────────────────────────────────────┐
  │  greentic-start  ── verify signature vs trust root       │
  │                  ── snapshot → apply → roll back on fail │
  └──────────────────────────────────────────────────────────┘
             v1  (webchat)   ⇄   v2  (webchat + telegram)

Install

Two binaries. Everything else — bash, curl, python3 — is already on every Linux and macOS box.

install 
$ cargo binstall greentic-deployer@1.1.10 greentic-start@1.1.9

Set up a workspace

A directory, the two bundles you'll switch between, and an isolated HOME so none of this touches your real ~/.greentic.

setup 
$ export DEMO=~/greentic-demo
$ mkdir -p $DEMO/bundles $DEMO/manifests $DEMO/home && cd $DEMO

# the content: v1 = webchat, v2 = webchat + telegram
$ REL=https://github.com/greenticai/greentic-cloud-update-demo/releases/download/content-v1
$ curl -fsSL -o bundles/v1.gtbundle        $REL/v1.gtbundle
$ curl -fsSL -o bundles/v1.gtbundle.sha256 $REL/v1.gtbundle.sha256
$ curl -fsSL -o bundles/v2.gtbundle        $REL/v2.gtbundle
$ curl -fsSL -o bundles/v2.gtbundle.sha256 $REL/v2.gtbundle.sha256

# every command below runs against this HOME, never your real one
$ export HOME=$DEMO/home

Use a throwaway terminal.

export HOME=… repoints this shell. That is what keeps the demo isolated — but every other command you run in this terminal also sees the fake home. Open a fresh terminal when you're done, or just close it.

The six steps

  1. Set the token

    Claim a private namespace on the plan server. You get back an unguessable environment id, an upload credential scoped to it, and the endpoint your runtime will poll. Reads need no credential at all.

    terminal 1 
    $ SERVER=https://greentic-plan-server.vladik-dobrik.workers.dev
$ curl -fsS -X POST $SERVER/v1/demo/session -o $DEMO/session.json

$ export GREENTIC_PLAN_UPLOAD_TOKEN=$(python3 -c "import json;print(json.load(open('$DEMO/session.json'))['upload_token'])")
$ export PLAN_ENDPOINT=$(python3 -c "import json;print(json.load(open('$DEMO/session.json'))['plan_endpoint'])")

$ echo $PLAN_ENDPOINT
https://greentic-plan-server.vladik-dobrik.workers.dev/v1/environments/demo-20f4d20e75e231da/plan

    GREENTIC_PLAN_UPLOAD_TOKEN is the one variable op updates publish reads. Passing it on the command line would leave it in your shell history and in ps.

    Now write the two manifests. This is the only place the endpoint appears:

    terminal 1 
    $ V1SHA=$(awk '{print $1}' bundles/v1.gtbundle.sha256)
$ V2SHA=$(awk '{print $1}' bundles/v2.gtbundle.sha256)

$ cat > manifests/v1.json <<EOF
{
  "schema": "greentic.env-manifest.v1",
  "environment": { "id": "local" },
  "updates": {
    "plan_endpoint": "$PLAN_ENDPOINT",
    "on_notify": "apply",
    "poll_interval_secs": 60
  },
  "bundles": [
    { "bundle_id": "updatedemo",
      "bundle_path": "$DEMO/bundles/v1.gtbundle",
      "bundle_digest": "sha256:$V1SHA" }
  ]
}
EOF

$ cat > manifests/v2.json <<EOF
{
  "schema": "greentic.env-manifest.v1",
  "environment": { "id": "local" },
  "bundles": [
    { "bundle_id": "updatedemo",
      "bundle_path": "$DEMO/bundles/v2.gtbundle",
      "bundle_digest": "sha256:$V2SHA" }
  ]
}
EOF

    Only v1.json carries an updates block. That block is the subscription — an environment that never declares one fetches nothing, ever.

  2. Start the env

    env init mints your operator key and seeds its public half into the environment's trust root. env apply deploys v1 and — because the manifest declares an updates block — subscribes to the channel. Then the runtime comes up and starts polling.

    terminal 1 — leave the last one running 
    $ greentic-deployer op env init
$ greentic-deployer op env apply --answers $DEMO/manifests/v1.json

$ greentic-start start --env local --no-browser

    greentic-start runs in the foreground. Its log is the demo — keep this terminal visible and run the rest in a second one (re-exporting DEMO, HOME, GREENTIC_PLAN_UPLOAD_TOKEN and PLAN_ENDPOINT there).

  3. Publish v2 to the server

    This reads the next sequence number off the server, signs the manifest in memory with your operator key, and uploads the bytes. The private key never leaves your machine, and the server never sees it.

    terminal 2 
    $ greentic-deployer op updates publish local --target-file $DEMO/manifests/v2.json

    Now do nothing. Nobody pushed anything at the runtime. Within one poll interval (≤60s) it fetches the plan, verifies the signature against its own trust root, snapshots, applies, and moves traffic — and terminal 1 says so:

    terminal 1 — the runtime's own log 
    update-notify: applied update plan `01KXCW2Q…` to env `local`
update-poll: env `local` plan sequence 1 -> 200 (applied)
runtime-config reloaded: 1 → 1 deployment(s), 1 → 1 revision(s)

  4. Check the config

    What is the environment actually serving right now? The pack list is the giveaway.

    terminal 2 — check config 
    $ greentic-deployer op env show local | python3 -c '
import json, sys
e = json.load(sys.stdin)["result"]["environment"]
live = {x["revision_id"] for s in e["traffic_splits"] for x in s["entries"] if x["weight_bps"] > 0}
for r in e["revisions"]:
    if r["revision_id"] in live:
        print("sequence", r["sequence"], r["bundle_digest"][:23])
        print("packs   ", ", ".join(p["pack_id"] for p in r["pack_list"]))'

sequence 2 sha256:feb45fafda62abe2
packs    app-webchat-bot, messaging-telegram   ← v2 is live

    And what the server is holding, plus the channel you subscribed to:

    terminal 2 
    $ curl -s $PLAN_ENDPOINT/meta
{"sequence":1,"plan_sha256":"5924ba178c47b29e…","uploaded_at":"2026-07-13T05:24:45Z"}

$ greentic-deployer op updates config-show local
    enabled=true  on_update=apply  poll_interval_secs=60

  5. Publish v1 — go back

    Same command, the other manifest. Sixty seconds later the environment is webchat-only again, live, with no restart.

    terminal 2 
    $ greentic-deployer op updates publish local --target-file $DEMO/manifests/v1.json

    Going back to v1 is not a downgrade.

    Anti-rollback protects the plan sequence, which only ever grows — 1, 2, 3, 4. The content a plan points at is free to move in either direction. So you can flip v1 ⇄ v2 as often as you like, while replaying an old signed plan stays impossible.

    v1.json also carries an updates block, and publish strips it before signing. A signed plan that could re-point the channel it arrived on would be a self-perpetuating takeover primitive, so it is not allowed to exist.

  6. Check the config again

    Same command as step 4.

    terminal 2 — check config 
    $ greentic-deployer op env show local | python3 -c '
import json, sys
e = json.load(sys.stdin)["result"]["environment"]
live = {x["revision_id"] for s in e["traffic_splits"] for x in s["entries"] if x["weight_bps"] > 0}
for r in e["revisions"]:
    if r["revision_id"] in live:
        print("sequence", r["sequence"], r["bundle_digest"][:23])
        print("packs   ", ", ".join(p["pack_id"] for p in r["pack_list"]))'

sequence 3 sha256:6e6f24f5c3f12c41
packs    app-webchat-bot                        ← back on v1

    The revision sequence went 1 → 2 → 3 while the content went v1 → v2 → v1. Repeat steps 3–6 as long as you like.

Clean up

cleanup 
# Ctrl-C the runtime in terminal 1, then close the terminals
# (they hold the fake HOME) and delete the directory:
$ rm -rf ~/greentic-demo

What this actually proves

ClaimWhat enforces it
The server can't forge an update Plans are signed by your operator key, which never left your machine. The server stores bytes and holds no key; a plan it invented would fail verification against your trust root.
An attacker on the wire can't either Plan reads are anonymous and unauthenticated on purpose. The signature is the trust anchor — not TLS, not the origin.
You can't be replayed an old version Every plan carries a monotonic sequence, and the anti-rollback check reads it from the signed plan — not from anything the server claims.
Two publishers can't fork a channel The store enforces strict monotonicity inside a Durable Object, so the read-modify-write is serialized. The loser of a race gets 409, never a silent fork.
Nothing is pushed at your runtime It polls. Discovery is not authority: no updates block, no fetching — ever.
A plan can't hijack the channel publish strips updates from a plan target before signing, and the apply path rejects one that smuggles it in.
A bad update rolls back Apply runs under a pre-apply snapshot, with post-apply verification and automatic rollback.

If something goes wrong

“port 8080 is already in use”

The runtime serves on the environment's listen address, and 8080 is the default. It is a property of the environment, not a flag on greentic-start — so the fix is to free the port. Find what holds it and stop that:

bash — Linux / macOS 
$ lsof -nP -iTCP:8080 -sTCP:LISTEN

A stray greentic-start from an earlier run is the usual culprit. Stop it with kill <pid> and start again.

Nothing happens after publish

Give it a full minute — 60s is the poll floor, and the runtime only checks on its own schedule. Then confirm the channel is actually on: greentic-deployer op updates config-show local should say enabled=true. If it doesn't, env apply was run with a manifest that had no updates block.

Your namespace expired

Namespaces on the shared server live 24 hours. Redo step 1 for a fresh one, rewrite the manifests (the endpoint changes), and re-run op env apply --answers $DEMO/manifests/v1.json so the environment picks up the new channel.

These are content-only plans, so your binaries are safe.

A plan can also carry a binary artifact, and the runtime that applies one replaces its own executable — which, here, would be the greentic-start on your PATH. Nothing above does that. If you go on to publish a --binary plan, run a copy of the runtime from somewhere else first.

Why is the upload endpoint open to anyone?

Because it buys an attacker nothing. Guess someone's namespace and the worst you can do is deny them an update: their runtime rejects any plan not signed by a key its trust root already carries. Denial, not compromise — which is exactly the property the whole design exists to show, so it would be dishonest to hide it behind a password.

Run your own plan server

The server is a small Cloudflare Worker; its source is in server/ in the repo. Deploy your own and point step 1 at it — nothing else about the run changes.

bash 
$ git clone https://github.com/greenticai/greentic-cloud-update-demo
$ cd greentic-cloud-update-demo/server && ./deploy.sh

# then, in step 1:
$ SERVER=https://your-worker.workers.dev
